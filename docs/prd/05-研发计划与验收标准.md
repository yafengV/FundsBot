# FundsBot PRD 05：研发计划与验收标准

> 文档目标：明确里程碑、交付件、测试策略、上线门槛。

## 1. 研发里程碑（8 周）

## M1（第 1-2 周）基础能力
- 完成：账本模型、手动持仓、首页总览静态版
- 交付件：数据模型文档、核心页面原型、基础接口

## M2（第 3-4 周）实时能力
- 完成：实时估值接入、持仓列表、基金详情
- 交付件：行情适配层、刷新机制、异常提示
- **tushare集成重点**:
  1. tushare客户端封装和配置管理
  2. 测试环境（token: `cb8bde72fbaaa...`）和生产环境（token: `d06bcb6694d4a...`）切换
  3. 基金基础信息、实时估值、历史净值接口对接
  4. 数据缓存和积分使用监控
  5. 数据源异常降级处理机制

## M3（第 5-6 周）增长能力
- 完成：OCR 导入草稿、净值提醒、周报月报
- 交付件：OCR 校验流程、提醒中心、报表模板

## M4（第 7-8 周）上线能力
- 完成：稳定性优化、埋点看板、灰度发布
- 交付件：监控告警、回滚预案、上线复盘模板

## 2. 测试策略

## 2.1 功能测试
- 覆盖：所有 P0 需求正/反向路径
- 要求：核心路径通过率 100%

## 2.2 回归测试
- 覆盖：导入、收益计算、通知触发、报表生成
- 要求：版本冻结后 2 轮回归无阻塞缺陷

## 2.3 数据正确性测试
- 构造：固定样例账本（已知理论结果）
- 验证：总资产、当日盈亏、累计盈亏误差在允许范围内

## 2.4 性能测试
- 首页首屏 <= 2s（常规网络）
- 500 条持仓列表滚动不卡顿

## 3. 验收标准（Go/No-Go）
1. 导入成功率 >= 85%
2. 净值提醒触达率 >= 98%
3. 核心计算指标误差 <= 0.1%
4. P0 阻塞缺陷为 0
5. 灰度用户负反馈率在阈值内

## 4. 上线与灰度策略
- 灰度 5% -> 20% -> 50% -> 100%
- 每阶段观察 24-48 小时核心指标
- 触发回滚条件：通知系统故障、收益计算异常、崩溃率激增

## 5. 上线后两周重点观察
- 用户是否完成首次导入
- 是否开启提醒并持续点击
- 周报是否被实际阅读
- 反馈中“数据不准/看不懂”的占比

## 6. 角色分工建议
- 产品：口径统一、需求验收
- 前端：首页、列表、详情、报表 UI
- 后端：导入、计算、提醒、报表任务
- 测试：回归矩阵 + 数据校验集
- 运营：通知文案、周报可读性优化
