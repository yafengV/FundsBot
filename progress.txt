## Codebase Patterns
- Settings frontend flows should allowlist datasource form fields (`env`, `reason`, `targetVersion`) before calling config services so plaintext token fields are dropped at the page-model boundary.
- Reminder-driven frontend navigation should resolve `deepLink` query params first and fallback to structured reminder payload fields before calling `getNavDailySummary`, so notification entry behavior stays consistent with backend summary routing.
- Database changes should ship with paired `*.up.sql` and `*.down.sql` files to keep rollbacks deterministic.
- Validate migration behavior with an actual SQLite apply/rollback test instead of only checking file existence.
- macOS default Bash is 3.2 in this environment, so avoid associative arrays in shared shell scripts.
- Shared finance metrics must use `shares_x10000` + NAV(4dp) -> cents via rounded integer math to keep dashboard/report/reconcile outputs consistent.
- Ledger list summaries should include only `holding` positions; `cleared` positions remain historical records but are excluded from F01 summary totals.
- For ledger-scoped reads, normalize with `switchLedgerScope` first and then reuse `getScopedPositions` plus `getScopeSummary` to keep single-ledger/all-ledger semantics aligned.
- Position write APIs should parse decimal `shares`/`costNav` into 4dp scaled integers (`*_x10000`) and derive `invested_cents` via rounded integer math.
- Position mutation flows should apply idempotency before any write and dedupe by `(position_id, idempotency_key)` to keep retries side-effect free.
- Read-side fallback should return a stable `{ items, meta }` shape with `meta.isFallback` and `meta.lastSuccessAt` so UI can degrade without changing render logic.
- CSV import should persist normalized `validated_rows` with batch metadata so commit can support partial success and deterministic checksum dedupe.
- OCR draft flows should persist per-row `uncertainFields` and require explicit `confirmedFields` coverage before confirm can write positions/txns.
- Quote fetch flows should retry retriable upstream failures up to 2 times, then fallback to cached values and surface per-type freshness (`fresh/stale/degraded`) metadata.
- Dashboard/position read responses should derive delay markers from the worst freshness in-scope (`degraded` > `stale` > `fresh`) and expose the latest `lastUpdatedAt` timestamp for UI hints.
- Notification trigger flows should dedupe by `(user_id, category, date)` before dispatch and downgrade external channels to `in_app` during do-not-disturb windows.
- External notification delivery should record per-attempt retry metadata (attempt index/backoff/status) so fallback and SLA metrics can be computed from persisted trigger logs.
- Report snapshots should be append-only per `(user, scope, period)` with monotonically increasing `version`; recalc writes a new version and read defaults to latest.
- Report fallback responses should only downgrade to the latest snapshot within the same `period_type + ledger scope` and include explicit fallback metadata (`reason`, requested period key, fallback timestamp).
- Reconcile tasks should compare only the fixed metric set (`totalAssetCents`, `dailyEstimatedPnlCents`, `dailyFinalPnlCents`, `cumulativePnlCents`) against expected sample baselines and classify by bps thresholds (`pass <= 10`, `warn <= 50`, else `fail`) with alert SLA tracking.
- Reconcile recalculation should append new `version` rows per `(run_date, ledger_scope, ledger_id)` and mark runtime fallback via `getLatestPassedReconcileVersion` instead of mutating historical rows.
- Datasource config switching should load tokens only via secret/env providers (`getSecret` or `process.env`) and persist only env/masked metadata, never plaintext tokens.
- Cross-module observability should wrap exported service handlers via `withObservedAction` so request IDs, structured logs, action metrics, trace spans, and threshold alerts stay consistent without duplicating instrumentation logic.
- Frontend feature delivery in this repo should encapsulate UI behavior as a pure page model + HTML renderer module so interaction flows can be validated in Node tests without introducing framework/runtime coupling.
- Import frontend flows should persist latest validation/draft batch context in page-model state so commit/confirm actions can default to the most recent batch without extra UI wiring.
- Frontend page-model query updates should be staged and rolled back on downstream service validation errors so error views render without poisoning state with invalid filters/sort/scope values.
- Report frontend flows should drive read and share actions from the same normalized query state (`periodType/scope/ledgerId/periodKey`) so exported artifacts match the currently viewed snapshot and fallback context.
